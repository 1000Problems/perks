# Architecture — Credit Card Recommender (Perks)

## Stack at a glance

| Layer | Choice |
|---|---|
| Framework | Next.js 15, App Router |
| Language | TypeScript |
| Styling | Tailwind CSS |
| UI primitives | shadcn/ui |
| Auth + DB | Supabase (Postgres + Auth) |
| Hosting | Vercel |
| Card data | Versioned JSON files in repo |
| Recommendation engine | Pure TypeScript, runs in browser |
| Schema validation | Zod |
| Forms | React Hook Form |
| Icons | Lucide React |
| Repo | GitHub, 1000Problems org |

## What runs where

**Browser does the heavy lifting.** Card data is bundled with the app (one fetch, then cached). The user's spend profile, card list, and trip plans live in React state during a session and persist to Supabase between sessions. The recommendation engine is a pure function: `(userProfile, cardData) → rankedRecommendations`. It runs every time the user changes a slider or toggles a setting, with no network round trip.

**Vercel does SSR for marketing and app shell.** The landing page, "/best-cards-for-X" content pages (later), and the app shell are server-rendered for SEO and first-paint speed. Once the user is in the app, interactivity is client-side.

**Supabase does identity and profile persistence.** Email/password auth with email verification disabled. A single `profiles` table holds the user's spend profile, card list, and trip plans as JSON columns. Row-level security restricts each user to their own row.

**No custom backend.** No Express, no FastAPI, no API routes beyond a few thin handlers for things that genuinely require server-side execution (e.g., a small endpoint to fetch the latest card data version). The Supabase JS client talks to the database directly from the browser, gated by RLS policies.

## Data layer

### Card database (static, versioned)

Lives in `data/*.json`. Generated by Claude Code research (see `RESEARCH_PROMPT_FOR_CODE.md`). Loaded once at app start, parsed and validated against Zod schemas in `lib/data/schema.ts`. Cached in memory for the session. Updates are a PR, a review, a deploy — no admin panel, no scraper.

Engine never imports raw JSON; it imports a `CardDatabase` object that's been schema-validated. If a card is malformed, build fails or app shows a warning at load time.

### User data (dynamic, Supabase)

Single `profiles` table. One row per user. Schema:

```sql
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  spend_profile jsonb not null default '{}',
  brands_used jsonb not null default '[]',
  cards_held jsonb not null default '[]',
  trips_planned jsonb not null default '[]',
  preferences jsonb not null default '{}',
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

alter table profiles enable row level security;

create policy "users read own profile" on profiles
  for select using (auth.uid() = id);
create policy "users update own profile" on profiles
  for update using (auth.uid() = id);
create policy "users insert own profile" on profiles
  for insert with check (auth.uid() = id);
```

JSON columns are deliberately schemaless at the DB level. Schema enforcement happens in TypeScript via Zod when reading/writing. This lets us evolve the profile shape without migrations during early development.

## Auth

**Email + password, email verification disabled.** Configured in Supabase dashboard: Authentication → Providers → Email → "Confirm email" set to off. UI labels the email field as "Account" since the user said "account and password," but Supabase requires a string-shaped-like-email. Any string with an `@` and a domain works; we'll add light client-side validation that it looks email-ish.

Login flow:
1. User enters email + password on `/login` or `/signup`
2. Supabase JS client calls `signInWithPassword` or `signUp`
3. Session token stored in HTTP-only cookies (Supabase handles this)
4. Server components read the session via `lib/supabase/server.ts` to gate the `(app)/` route group
5. Client components use `lib/supabase/client.ts` for everything else

No social login, no magic links, no MFA in v1. Add later if needed.

**Password requirements**: minimum 8 characters, no other rules. Don't make signup friction in a tool nobody's heard of yet.

## The recommendation engine

The engine is a pure function. No side effects, no async, no network. This is the most important property of the architecture — it's what makes the UI feel instant and what makes the engine trivially testable.

```ts
function recommend(
  user: UserProfile,
  cards: CardDatabase,
  options: ScoringOptions
): RankedRecommendation[]
```

Inside, it runs five passes:

1. **Filter eligible cards.** Apply 5/24, once-per-product, etc. given the user's stated card history. Excluded cards get an `eligibility: 'denied'` flag, not removed (we want to surface them as "you're not eligible for this right now" if useful).
2. **Score each card.** For each candidate, compute marginal annual value: (rewards on user's spend with this card added) - (annual fee) - (unredeemed credits, valued realistically) - (already-covered perks discounted).
3. **Compute deltas.** Subtract the user's current wallet's value to get the marginal contribution of adding this card.
4. **Generate "why" sentences.** Look at where the delta came from — biggest spend category improvement, biggest perk gain, biggest sweet spot unlocked — and build a one-line plain-English explanation.
5. **Rank and return top N.** Sort by configurable criterion (delta value, no-AF only, premium tier).

Each pass is its own file in `lib/engine/`. Each file exports pure functions with explicit inputs and outputs. Unit tests live in `tests/engine/` and run on every commit.

The engine is reactive: any state change in the UI (spend slider, year-1 toggle, ranking criterion) triggers a re-run. Because it's pure and fast (well under 50ms for 150 cards), this works without memoization tricks. Add `useMemo` only if profiling shows we need it.

## Mobile vs desktop IA

Same routes, same data, different layout primitives.

**Desktop**: Three-column recommendation panel as designed. Fixed sidebar for navigation.

**Mobile**: The three columns become a horizontal swipe (your wallet → top 5 cards → drill-in detail), or alternatively a bottom-sheet pattern where tapping a card raises the detail. Designer's call. Routes are the same; the layout component picks the right primitive based on viewport.

We use Tailwind responsive classes (`md:`, `lg:`) for layout switches. No separate mobile codebase, no separate routes.

## Deployment

GitHub repo at `github.com/1000Problems/perks` (matches the org pattern from `1000p-deploy-v2`). Vercel project linked to the repo. Pushes to `main` deploy to production; PRs deploy preview environments automatically.

Custom subdomain `perks.1000problems.io` (or similar — final name TBD), SSL handled by Vercel.

Supabase project is a separate concern, set up once via dashboard. Production project for the live site, plus a "dev" project for local development.

## Environment variables

All secrets live in Vercel project settings (production) and `.env.local` (gitignored, local dev). Names only:

- `NEXT_PUBLIC_SUPABASE_URL` — Supabase project URL, public-safe
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` — Supabase anon key, public-safe (RLS protects data)
- `SUPABASE_SERVICE_ROLE_KEY` — server-only, used for any admin operations (deferred for now, but reserve the name)

No third-party API keys in v1. No Plaid, no email service (since we're not verifying emails), no analytics in the architecture (add later via env-gated Vercel Analytics if wanted).

## What we're deliberately leaving out

- **Statement upload / Plaid** — deferred. Architecture supports adding it later as a separate route and a `lib/transactions/` module. No changes needed to the core engine — it just gets a better-quality `spend_profile` input.
- **Affiliate links and disclosure UX** — deferred until monetization. Card schema already has a `sources` array; affiliate URLs slot in alongside.
- **Card data update automation** — deferred. Manual PR-based updates are fine while we're building.
- **Email verification, password reset flows** — deferred. We'll add when user feedback says they need it.
- **Analytics, error tracking** — deferred. Add Vercel Analytics + Sentry later via env vars.
- **Server-side recommendation API** — deferred. Only needed if we want to A/B test scoring weights or run the engine for users who don't have JS enabled. Neither is a v1 concern.

## Risks worth naming

1. **Bundle size.** 150 cards × full perk descriptions × destination data could push the JSON to 1-2 MB. We'll watch it. If it grows, we split the data: ship a slim version on first load, fetch detail on demand when the user clicks a card.
2. **Card data freshness.** Manual updates means data goes stale. Mitigation: a "last verified" date on each card, prominent in the UI, and a clear "data updated YYYY-MM-DD" stamp at the bottom of the rec panel. Trust through transparency.
3. **Supabase free tier limits.** 50K monthly active users, 500MB DB. We're nowhere near this. Reassess if we hit it.
4. **Vercel bandwidth.** Free tier is 100GB. Card art images could push this if we get traffic. Mitigation: serve from Cloudflare Images or a CDN bucket later.

## Build order

1. Scaffold the Next.js project, Vercel deploy, Supabase auth working (login/signup/session).
2. Card data schemas, loader, validation. Stub data with 5-10 cards before Code's research is complete.
3. Onboarding flow screens 1-3, persisting to Supabase.
4. Recommendation engine, fully tested in isolation against stub data.
5. Recommendation panel UI (Screen 4), wired to engine.
6. Trip planner.
7. Polish, mobile layouts, edge cases, empty states.

Each step is mergeable on its own. The engine being a pure function means we can build and test it before the UI is ready, and vice versa.
